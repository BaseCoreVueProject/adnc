# ADNC Inter-service Communication: gRPC

[GitHub Repository](https://github.com/alphayu/adnc)

In ADNC, synchronous inter-service calls can also be implemented using gRPC. gRPC offers better performance and stronger contracts (via `.proto` files), making it ideal for internal communication. This guide uses `GrpcClientDemoController.cs` as an example.

---

## 0. Quick Start (4 Steps)

1. **Define the `.proto`**: Place it in a shared directory (e.g., `src/Demo/Shared/protos/services/admingrpc.proto`).
2. **Generate Client Code**: The caller references the `Remote.Grpc` project, which automatically generates the gRPC Client from `.proto` files.
3. **Register the Client**: Call `AddGrpcClient<...>(...)` in the caller's dependency registration.
4. **Invoke**: Inject the generated `*GrpcClient`, construct the Request + Metadata (e.g., `GrpcClientConsts.BasicHeader`), and call the method.

## 1. Design Recommendations

- **Prefer for Queries/Validations**: Minimize long synchronous chains. Use Event-driven patterns for operations requiring eventual consistency.
- **Unified Contracts**: Always generate code from shared `.proto` files. Avoid manual DTO alignment.
- **Governance**: Apply standard timeout, retry, and circuit breaker policies to prevent thread pool exhaustion.

## 2. Project Structure

```
src/Demo/Shared/protos/        # Shared Proto definitions
src/Demo/Shared/Remote.Grpc/   # Client code generation project
```

The **Server side** (the service providing the gRPC endpoint) implements the `*GrpcBase` generated by the proto and maps it to an endpoint.

## 3. Defining the gRPC Service

```proto
service AdminGrpc {
  rpc GetSysConfigList(SysConfigSimpleRequest) returns (SysConfigListReply);
}
```

## 4. Port and Kestrel Configuration (HTTP/2)

gRPC requires HTTP/2. ADNC demo services typically expose a dedicated gRPC port:
- **Admin**: `50011`
- **Cust**: `50031`

Example `appsettings.json`:
```json
"Kestrel": {
  "Endpoints": {
    "Default": { "Url": "http://0.0.0.0:50010" },
    "Grpc": { "Url": "http://0.0.0.0:50011", "Protocols": "Http2" }
  }
}
```

## 5. Registering the gRPC Client

Register the client in the application layer. The `AddGrpcClient` implementation handles address resolution (Direct/Consul/CoreDNS) and load balancing (RoundRobin).

```csharp
var policies = GenerateDefaultGrpcPolicies();
AddGrpcClient<AdminGrpc.AdminGrpcClient>(ServiceAddressConsts.AdminDemoService, policies);
```

## 6. Authentication via Metadata

In gRPC, authentication is passed via `Metadata`. ADNC uses marker headers to let the framework inject the actual token:
- `GrpcClientConsts.BasicHeader`: Sets `Authorization: Basic`.
- `GrpcClientConsts.BearerHeader`: Sets `Authorization: Bearer`.

The framework's `TokenDelegatingHandler` automatically completes the token (e.g., `Authorization: Basic {token}`).

## 7. Fault Tolerance (Polly)

gRPC calls reuse the same fault tolerance policies as HTTP:
- **Retry**: On timeouts or 5xx errors.
- **Timeout**: Maximum duration for a call.
- **Circuit Breaker**: Prevents failure propagation.
